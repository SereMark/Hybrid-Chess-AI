# Hybrid Chess AI

[![CI](https://github.com/SereMark/Hybrid-Chess-AI/actions/workflows/ci.yml/badge.svg)](https://github.com/SereMark/Hybrid-Chess-AI/actions/workflows/ci.yml)

Hybrid Chess AI is my attempt at building a small AlphaZero-style chess system from the ground up. The project combines a fast C++ chess engine and Monte Carlo Tree Search (MCTS) with a PyTorch policy/value network. Python handles the parts that are easier to experiment with, while C++ handles the search work that needs to run many times per second.

This is not meant to compete with Stockfish, and it is not a finished chess product. It is mainly a research and engineering project where I could explore reinforcement learning, game-tree search, native Python extensions, and the practical problems that appear when all of those pieces have to work together.

## What is included?

- A C++23 chess core with legal move generation, FEN support, make/unmake, draw detection, and position hashing.
- A batched MCTS implementation that reuses its tree between moves.
- Python bindings built with `pybind11`.
- A PyTorch residual network with separate policy and value heads.
- Parallel self-play, a replay buffer, training, checkpointing, and resume support.
- Arena matches for deciding whether a new model should replace the current best model.
- Benchmark scripts and saved reports for the main performance-sensitive parts.
- Tests for both the native chess code and the Python training pipeline.

The included `opening_book.json` was generated by merging the ECO-letter TSV files from
[`lichess-org/chess-openings`](https://github.com/lichess-org/chess-openings), a CC0/public-domain dataset of chess
opening names and positions.

## How the project is organised

```text
src/core/           C++ chess rules, MCTS, and pybind11 bindings
src/python/         neural network, self-play, training, evaluation, and utilities
configs/            five experiment configurations with different search/training trade-offs
tests/              pytest tests for the C++ extension and Python code
tools/              benchmark commands
benchmark_reports/  selected CSV and JSON results from completed runs
docs/               setup, architecture, testing, and benchmark notes
```

Training runs are written to `runs/`. They can become very large, so checkpoints, PGNs, caches, and raw run data are intentionally not committed.

## How it works

A training iteration follows the same basic loop as AlphaZero:

1. The current network plays games against itself.
2. MCTS improves the network's move probabilities during each game.
3. Positions, visit counts, and final game results are stored in the replay buffer.
4. The network trains on samples from that buffer.
5. At fixed intervals, the updated model plays an arena match against the retained best model.
6. If the candidate scores well enough, it becomes the new best model.

The main implementation split is deliberate. Chess state and MCTS live in C++ because they are called constantly during self-play. Training, experiment configuration, metrics, and checkpoint management stay in Python because those parts change more often and benefit from the PyTorch ecosystem.

For a deeper explanation, see [the architecture notes](docs/architecture.md).

## Requirements

The project has mainly been developed and tested with:

- Windows 11 x64
- Python 3.13
- Visual Studio 2022 Build Tools with the **Desktop development with C++** workload
- CMake 3.21 or newer
- An NVIDIA CUDA-capable GPU for realistic training runs

CPU-only smoke tests work, but self-play and training are much slower. The supplied experiment configs were made around an RTX 3070 Mobile and Ryzen 7 5800H, so they may need adjusting for other hardware.

## Quick start

Create a virtual environment and install the project:

```powershell
py -3.13 -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
python -m pip install -e ".[dev]"
```

Build the native extension:

```powershell
cmake -S . -B build -G "Visual Studio 17 2022" -A x64
cmake --build build --config Release
```

CMake copies the compiled `chesscore` module into `src/python/`. Once that is available, run the tests:

```powershell
$env:PYTHONPATH = "$PWD\src\python"
python -m pytest
```

More detailed setup and troubleshooting notes are in [docs/setup.md](docs/setup.md).

## Running an experiment

Start with the baseline configuration:

```powershell
$env:PYTHONPATH = "$PWD\src\python"
hybrid-chess-train -c configs\run1_baseline.yaml
```

Useful alternatives include:

- `run2_deep_thought.yaml` — spends more MCTS simulations on each move.
- `run3_rapid_fire.yaml` — searches less deeply but produces games faster.
- `run4_high_exploration.yaml` — adds more exploration during self-play.
- `run5_aggressive_updates.yaml` — trains more frequently from fewer generated games.

Use `--device cpu`, `--device cuda`, or `--device cuda:0` to choose a device. Add `--resume` to continue from the newest checkpoint that can be found:

```powershell
hybrid-chess-train -c configs\run1_baseline.yaml --device cuda --resume
```

Each run keeps its merged configuration, metrics, checkpoints, and arena PGNs together under `runs/<run-id>/`.

## Tests and code quality

The local checks used by CI are:

```powershell
$env:PYTHONPATH = "$PWD\src\python"
python -m pytest
python -m ruff check src/python tests
python -m black --check src/python tests
python -m mypy
```

Build `chesscore` before running the full suite. Native-dependent tests are skipped when the extension is missing, which is useful for a quick Python-only check but is not the same as a complete test run.

See [docs/testing.md](docs/testing.md) for the areas covered by the tests.

## Benchmarks

The repository includes saved results for move generation, MCTS scaling, neural-network inference, training cost, and model ranking. These results are useful for comparing parts of this project, but they should not be treated as universal hardware-independent numbers.

The report list and reproduction commands are in [docs/benchmarks.md](docs/benchmarks.md).

## Current limitations

- The training setup is small compared with serious chess projects, so playing strength is limited by compute and data.
- Windows is the only platform covered by CI at the moment.
- The arena uses a relatively small number of games, which makes promotion decisions noisy.
- Most tuning was done for one laptop, so the default worker counts and batch sizes may not suit every machine.
- This is a research codebase rather than a server, GUI, or ready-to-use chess application.

These limitations are part of the project rather than something I want to hide. A large part of the work was learning where a clean idea on paper becomes messy once performance, reproducibility, and limited hardware are involved.
